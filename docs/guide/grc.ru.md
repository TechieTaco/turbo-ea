# GRC

Модуль **GRC** объединяет Управление, Риск и Соответствие в единое рабочее пространство по адресу `/grc`. Он консолидирует задачи, ранее разнесённые между «Поставкой EA» и TurboLens, чтобы архитектор, владелец риска и проверяющий по соответствию работали на общей основе.

В GRC три вкладки:

- **Управление** — Принципы EA и Architecture Decision Records (ADR).
- **Риск** — [Реестр рисков](risks.md) согласно TOGAF этапа G.
- **Соответствие** — сканер по запросу (CVE + анализ нормативных пробелов), который ранее жил в TurboLens.

К любой вкладке можно перейти прямой ссылкой через `/grc?tab=governance`, `/grc?tab=risk` или `/grc?tab=compliance`.

![GRC — вкладка «Управление»](../assets/img/ru/52_grc_upravlenie.png)

## Управление

Две панели рядом:

- **Принципы** — браузер только для чтения с EA-принципами, опубликованными в метамодели (формулировка, обоснование, последствия). Каталог редактируется в «Администрирование → Метамодель → Принципы».
- **Решения** — Architecture Decision Records. Каждый ADR фиксирует статус, контекст, решение, рассмотренные альтернативы и последствия. Решения, выпущенные мастером TurboLens Architect, попадают сюда черновиками для утверждения.

## Риск

![GRC — Реестр рисков](../assets/img/ru/53_grc_reestr_riskov.png)

Встраивает **Реестр рисков** TOGAF этапа G. Полный жизненный цикл, рабочий процесс статусов, переключатели матрицы и поведение владельцев описаны в [руководстве по Реестру рисков](risks.md). Самое существенное:

- Реестр живёт по адресу `/grc?tab=risk` (ранее находился в «Поставке EA»).
- Риски можно создавать вручную или **продвигать** из находки CVE либо из находки соответствия на вкладке «Соответствие».
- Продвижение идемпотентно — после продвижения находки её кнопка становится «Открыть риск R-000123».

## Соответствие

![GRC — сканер соответствия](../assets/img/ru/54_grc_sootvetstvie.png)

Сканер безопасности по запросу из двух независимых половин:

- **CVE-скан** — обращается в NIST NVD по вендорам / продуктам / версиям актуального ландшафта, после чего LLM приоритизирует находки.
- **Скан соответствия** — анализ пробелов по регулятиве на основе ИИ против включённых регулятив. Шесть фреймворков включены по умолчанию (EU AI Act, GDPR, NIS2, DORA, SOC 2, ISO 27001); администратор может включать и выключать их — а также добавлять собственные регулятивы вроде HIPAA или внутренних политик — в [**Администрирование → Метамодель → Регулятивы**](../admin/metamodel.md#compliance-regulations).

Находки **устойчивы между повторными сканами** — пользовательские решения, заметки рецензента, ИИ-вердикт пользователя по карточке и обратная ссылка на продвинутый Риск переживают последующие сканы. Находка, которую очередной проход больше не сообщает, помечается `auto_resolved` и по умолчанию скрывается; ранее продвинутый Риск сохраняется, чтобы аудиторский след не разрывался.

Сетка «Соответствие» зеркалит сетку «Инвентарь»: боковая панель фильтров с переключателями видимости столбцов, сохранённая сортировка, полнотекстовый поиск и панель деталей, отображающая жизненный цикл соответствия как горизонтальную фазовую шкалу:

```
new → in_review → mitigated → verified
                      ↘ accepted          (требует обоснования)
                      ↘ not_applicable    (пересмотр области)
                      ↘ risk_tracked      (устанавливается автоматически при продвижении в Риск)
```

С `security_compliance.manage` отметка флажка в шапке выполняет **фильтр-зависимый выбор всего**, после чего закреплённая панель инструментов позволяет применить **Изменить решение** (батч-переход) или **Удалить** выбранные находки. Недопустимые переходы сообщаются построчно в сводке частичного успеха, поэтому одна ошибочная строка не валит весь батч. Полный справочник действий — в [TurboLens → Безопасность и соответствие](turbolens.md#bulk-actions-on-the-compliance-grid).

Когда Риск, продвинутый из находки, закрывается или принимается, это **автоматически возвращается на находку** — связанная строка соответствия переходит на `mitigated` / `verified` / `accepted` / `in_review`, чтобы оставаться синхронной, без ручного сопровождения.

### Соответствие на одной карточке

Карточки, попавшие в область скана соответствия, также показывают вкладку **Соответствие** на своей странице деталей (управляется правом `security_compliance.view`). Она перечисляет каждую находку, в настоящее время привязанную к карточке, с теми же действиями «Подтвердить / Принять / **Создать риск** / **Открыть риск**», что и в GRC, — так Application Owner может сортировать свои находки, не покидая карточку.

## Разрешения

| Разрешение | Роли по умолчанию |
|------------|-------------------|
| `grc.view` | admin, bpm_admin, member, viewer |
| `grc.manage` | admin, bpm_admin, member |
| `risks.view` / `risks.manage` | см. [Реестр рисков § Разрешения](risks.md) |
| `security_compliance.view` / `security_compliance.manage` | см. [TurboLens § Security & Compliance](turbolens.md) |

`grc.view` управляет видимостью самого маршрута GRC — без него запись в верхнем меню скрыта. Каждая вкладка дополнительно обеспечивает собственное доменное разрешение, так что просматривающий, например, может читать реестр, но не может запускать LLM-скан.
